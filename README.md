# Food Delivery ETA Prediction ‚Äî Machine Learning Project

Predict the **delivery time** (`Time_taken (min)`) of a food order from operational, environmental, and
delivery-related information, using a Zomato-style dataset of **45,584 orders**.

The project is built in two stages:

| Stage | Deliverable | Models |
|-------|-------------|--------|
| **1 ‚Äî Baseline pipeline** | [`food_delivery_eta.py`](food_delivery_eta.py) | Linear Regression vs Random Forest |
| **2 ‚Äî Accuracy upgrade** | [`Food_Delivery_ETA_Ensemble.ipynb`](Food_Delivery_ETA_Ensemble.ipynb) | Random Forest **+** XGBoost ensemble |

Two rules are enforced everywhere:

1. **No data leakage** ‚Äî every transform that *learns* from data is fit on the **training split only**.
2. **No overfitting** ‚Äî models are regularised, cross-validated, and their **train-vs-test gap** is reported.

## Objective

Given the details of an order (rider, restaurant/customer location, weather, traffic, time, etc.), predict how
many minutes the delivery will take. We compare a simple linear baseline against progressively stronger
tree-based models and quantify how much each step actually helps.

## Feature Engineering

All engineered features are **row-wise** (each row uses only its own values or fixed constants), so they are
leakage-free and computed before the train/test split.

| Feature | Description |
|---------|-------------|
| `distance_km` | Haversine distance between restaurant and customer. Coordinates are `abs()`-corrected (India is +lat/+lon) and capped at 30 km to neutralise bad GPS values. |
| `prep_time_min` | Pickup time ‚àí order time, in minutes, with midnight wrap-around handled; implausible values (>120 min) dropped. |
| `order_hour`, `pickup_hour` | Hour of day extracted from the time fields. |
| `is_peak_hour` | 1 if ordered during the lunch (12‚Äì14) or dinner (19‚Äì22) rush. |
| `time_of_day` | Categorical part-of-day: Morning / Afternoon / Evening / Night. |
| `order_day_of_week`, `is_weekend` | Calendar signals from the order date. |
| `Delivery_person_Ratings` | Clipped to the valid **1‚Äì5** range. |

**Final feature set** ‚Äî 10 numeric + 8 categorical:

- **Numeric:** `Delivery_person_Age`, `Delivery_person_Ratings`, `multiple_deliveries`, `distance_km`,
  `prep_time_min`, `order_hour`, `pickup_hour`, `order_day_of_week`, `is_weekend`, `is_peak_hour`
- **Categorical (one-hot):** `Weather_conditions`, `Road_traffic_density`, `Vehicle_condition`, `Type_of_order`,
  `Type_of_vehicle`, `Festival`, `City`, `time_of_day`

Columns dropped as non-predictive / leakage-prone: `ID`, `Delivery_person_ID`, raw coordinates, raw date/time
strings.

## The Models ‚Äî How They Work

### 1. Linear Regression *(Stage 1 baseline)*
Fits a straight-line relationship: `ETA = w1*feature1 + w2*feature2 + ... + b`. It assumes each feature has a
constant, additive effect on delivery time. Features are **standardised** (`StandardScaler`) first so
coefficients are comparable. It is fast and interpretable but **cannot model interactions or non-linearities**
(e.g. "distance matters much more when traffic is *Jam*"), which caps its accuracy. Used as the reference point
to prove the value of the tree models.

### 2. Random Forest *(Stage 1 main + Stage 2 base)*
An ensemble of many **decision trees** trained by **bagging**:
- Each tree is trained on a random bootstrap sample of the rows and a random subset of the features
  (`max_features`).
- Every tree overfits *differently*; **averaging their predictions cancels out that noise**.
- This is **variance reduction** ‚Äî the forest is far more stable than any single tree.

It captures non-linearities and feature interactions automatically and needs no feature scaling. We regularise
it (`max_depth`, `min_samples_leaf`, `max_features < 1.0`) and tune with cross-validation to keep the
train/test gap small. **On this dataset it is the strongest single model.**

### 3. XGBoost *(Stage 2 base)*
Gradient-boosted decision trees built by **boosting**:
- Trees are added **sequentially**; each new tree is trained to correct the **residual errors** of the trees so
  far, scaled by a small `learning_rate`.
- This is **bias reduction** ‚Äî the model keeps refining where it is currently wrong.
- Regularised with row/column subsampling (`subsample`, `colsample_bytree`) and `reg_lambda`; the number of
  trees can be chosen via early stopping.

Because boosting learns *differently* from bagging, XGBoost's errors are not identical to the forest's ‚Äî which
is exactly what makes them useful to combine.

### 4. The Ensemble *(Stage 2 final)*
Random Forest (variance-focused) and XGBoost (bias-focused) make **partially independent errors**, so blending
them generalises better than either alone. Two blending strategies are used:

- **Weighted average** ‚Äî `final = w * RF + (1 - w) * XGB`. The weight `w` is **not guessed**: it is chosen by
  sweeping values and minimising error on **out-of-fold cross-validation predictions** (never the test set).
- **Stacking** ‚Äî a small **Ridge regression** meta-model *learns* the best combination of the two base
  predictions, also trained on out-of-fold predictions.

```
   Random Forest --+
                   +-->  Weighted Average  --+
   XGBoost --------+                         +-->  Final ETA
                   +-->  Ridge Stacking  ----+
   (weights/meta-model chosen by cross-validation -> no test leakage)
```

## Design Principles: No Leakage, No Overfitting

**No leakage ("no pre-feeding").** Imputation, one-hot encoding, and scaling are wrapped inside each model's
scikit-learn `Pipeline`/`ColumnTransformer`. During cross-validation they are **refit on each fold's training
data only**, so information from the validation/test rows can never influence training. Ensemble weights and the
stacking meta-model are selected on **out-of-fold predictions**, keeping the test set completely untouched until
the final evaluation.

**No overfitting.** Both tree models are regularised and tuned with `RandomizedSearchCV`. We report the
**train-vs-test R¬≤ gap** for every model, and confirm that **cross-validation MAE ‚âà test MAE**, proving the
reported accuracy is a genuine estimate of real-world performance.

## Results

### Stage 1 ‚Äî Linear Regression vs Random Forest (test set)

| Model | MAE | RMSE | R¬≤ | Train‚ÜíTest R¬≤ gap |
|-------|----:|-----:|---:|:-----------------:|
| Linear Regression | 4.806 | 6.040 | 0.586 | 0.001 |
| Random Forest *(default, unregularised)* | 3.179 | 4.004 | 0.818 | 0.157 *(overfit)* |
| **Random Forest *(tuned)*** | **3.139** | **3.929** | **0.825** | **0.037** |

Random Forest cuts the error nearly in half vs the linear baseline, and regularisation shrinks the overfitting
gap from 0.157 to 0.037 while keeping accuracy.

### Stage 2 ‚Äî RF + XGBoost Ensemble (test set)

| Model | MAE | RMSE | R¬≤ |
|-------|----:|-----:|---:|
| Random Forest | 3.125 | 3.931 | 0.8248 |
| XGBoost | 3.141 | 3.955 | 0.8227 |
| Weighted Ensemble (RF 0.55 / XGB 0.45) | 3.114 | 3.916 | 0.8262 |
| **Stacking (Ridge meta-model)** | **3.114** | **3.915** | **0.8263** |

The ensemble is the best model on every metric. Overfitting stays controlled (RF gap 0.072, XGB gap 0.052; CV
MAE ‚âà test MAE).

## Reusing the Saved Models

Both base models are **full pipelines**: pass a DataFrame with the raw engineered feature columns and they handle
imputation/encoding internally.

**Stage 1 ‚Äî Random Forest:**
```python
import joblib
rf = joblib.load("outputs/models/random_forest_pipeline.joblib")
predictions = rf.predict(X_new)     # X_new: DataFrame with the engineered feature columns
```

**Stage 2 ‚Äî Ensemble:**
```python
import joblib, numpy as np

bundle  = joblib.load("outputs_ensemble/models/ensemble_bundle.joblib")
rf, xgb = bundle["rf"], bundle["xgb"]
w       = bundle["rf_weight"]        # e.g. 0.55
stacker = bundle["stacker"]

rf_pred, xgb_pred = rf.predict(X_new), xgb.predict(X_new)

weighted_pred = w * rf_pred + (1 - w) * xgb_pred          # weighted-average ensemble
stacked_pred  = stacker.predict(np.c_[rf_pred, xgb_pred]) # Ridge stacking ensemble
```

## Key Findings

- **Non-linearity is the big win.** Moving from Linear Regression (R¬≤ 0.586) to Random Forest (R¬≤ 0.825) nearly
  halves the error ‚Äî traffic, distance, weather, and rider factors interact in ways a linear model cannot
  capture.
- **The ensemble is the best model, but the gain over a tuned Random Forest is small** (MAE 3.139 ‚Üí 3.114).
  Random Forest is already close to this dataset's predictive ceiling; the remaining error is largely
  **irreducible noise** (no feature separates, say, a 26-minute delivery from a 29-minute one). XGBoost,
  HistGradientBoosting, and extra geographic features were all tested and did not beat it.
- **The value here is in the method, not model complexity:** clean feature engineering, strict leakage
  prevention, and honest cross-validated evaluation. A materially more accurate model would require **richer
  data** (real GPS traces, live traffic, restaurant prep logs) rather than a fancier algorithm.
